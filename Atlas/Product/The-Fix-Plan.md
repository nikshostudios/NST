- Supabase Postgres — single source of truth, schema deployed, 638 requirements + 2,300+ candidates
- `/ai-agents/` layer — deployed on Railway, reachable from web app (Issue 1 fixed)
- Internal DB querying — works, returns candidates, free, instant
- `upsert_candidate_by_name()` — deployed, handles candidates without email
- Pipeline endpoint — works with 50-req limit (Issue 2 fixed)
- Dashboard frontend — React + Vite prototype exists at localhost:3000, dark theme, all pages scaffolded
- MCF market intelligence — returns job postings (useful for demand analysis, not sourcing)

**What's broken:**
- Foundit Recruiter — cookie auth approach is dead. Cookies expire every 24-48h, Akamai blocks Firecrawl, direct API interception failed (still returns 401). **This channel is gone for now.**
- Internal DB skill matching — composite skill strings ("ServiceNow JavaScript ITSM") don't match individual skills ("ServiceNow"). The `.overlaps()` query is fundamentally broken for this data shape.
- Apollo — no API key set. Free plan locks People Search. **$49/mo fixes this instantly.**
- MCF, Foundit Public, Scrape.do — all return job postings, not candidates. Removed from sourcing pipeline (Prompt 5 done).
- JD skills extraction — silently failing on requirement creation (Issue 9 not investigated).
- Dashboard — looks great but nothing is interactive. Every click is dead. Mock data only.
- Pending git commits — `upsert_candidate_by_name`, pipeline count fix, pipeline query limit not pushed to GitHub.

**What we learned from competitors:**
- Juicebox's "magic" is 7 layers: search parser → multi-source fetcher → LLM scorer → enrichment → sequencing → analytics → shortlist/export. Each layer is buildable.
- The gap nobody fills: powerful AI + agency-native workflow. Recruiterflow has workflow but weak AI. Juicebox has AI but no agency workflow.
- Per-requirement pricing (GoPerfect model) beats per-seat for agencies.
- Natural language search over your own data is the highest-impact feature — it's an LLM prompt + structured output + database query.

### Day 3: Build the Search Parser + LLM Scorer (Shoham)

**The problem (from technical-architecture-stolen, Layers 1+3):** Candidates exist in the DB but there's no intelligent way to find and rank them. The current matching is keyword overlap, which is why it fails on composite strings and misses semantic matches.

**Prompt for Conductor:**
```
Build two new modules in ai-agents/:

MODULE 1: ai-agents/search_parser.py
- Function: parse_search_query(natural_language: str) -> dict
- Takes natural language like "Senior Java developer, 5+ years, Singapore, fintech preferred"
- Uses Claude Haiku to extract structured JSON:
  {
    "hard_filters": {"title_keywords": [...], "location": "...", "min_years": N, "must_have_skills": [...]},
    "soft_criteria": [{"criterion": "...", "weight": "required|preferred"}]
  }
- Validate the JSON schema before returning

MODULE 2: ai-agents/candidate_scorer.py
- Function: score_candidates(candidates: list[dict], criteria: list[dict]) -> list[dict]
- Takes a list of candidate profiles and soft criteria
- Batches candidates in groups of 15
- Sends each batch to Claude Sonnet with prompt:
  "Score each candidate 0-100 on each criterion. Return JSON with per-criterion scores and one-line explanations."
- Parses response, calculates weighted overall score
- Returns candidates sorted by score, each with:
  {
    "candidate_id": "...",
    "overall_score": 85,
    "criteria_scores": [
      {"criterion": "Java experience", "score": 95, "explanation": "8 years Java including Spring Boot"},
      ...
    ]
  }
- Cache scores in a Supabase match_scores table (requirement_id, candidate_id, scores_json, scored_at)

MODULE 3: ai-agents/smart_search.py
- Function: smart_search(query: str) -> list[dict]
- Chains everything together:
  1. parse_search_query(query) -> filters + criteria
  2. Query Internal DB with hard_filters
  3. Query Apollo with hard_filters (in parallel with step 2)
  4. Merge + deduplicate results
  5. score_candidates(merged_results, criteria)
  6. Return ranked list

Add a FastAPI endpoint: POST /api/search with body {"query": "..."} that calls smart_search and returns results.
Test with: "Senior ServiceNow developer, 5 years, India, ITSM certification preferred"
```

**After Day 3:** A recruiter can type a natural language query and get back ranked, scored candidates from Internal DB + Apollo. This is the Juicebox experience, built on our data.

### Day 5: Build the Sequencing Engine (Shoham)

**The problem (from technical-architecture-stolen, Layer 5):** We have no way to do bulk personalised outreach. Juicebox lets you select 10 candidates and generate personalised emails for all of them in seconds.

**Prompt for Conductor:**
```
Build an email sequencing system in ai-agents/:

1. ai-agents/sequencing.py:
   - Function: generate_sequence_emails(candidates: list[dict], template: str, role: str) -> list[dict]
   - For each candidate, use Claude Haiku to personalise the email template using their profile data
   - Template uses merge fields: {{candidate_name}}, {{relevant_experience}}, {{role_title}}, {{company}}
   - Return list of {candidate_id, subject, body, personalisation_notes}

2. ai-agents/sequence_scheduler.py:
   - Store sequences in Supabase: sequences table (id, name, requirement_id, status, steps_json, created_at)
   - Store individual emails: sequence_emails table (id, sequence_id, candidate_id, step_number, subject, body, status, scheduled_for, sent_at, opened_at, replied_at)
   - Function: schedule_sequence(sequence_id) — sets scheduled_for timestamps (Day 0, Day 3, Day 7)
   - Function: get_pending_emails() — returns emails where scheduled_for < now and status = 'pending'
   - IMPORTANT: emails are set to status='draft' by default. A human must approve before sending.

3. Create Supabase tables: sequences, sequence_emails (add migration SQL)

4. FastAPI endpoints:
   - POST /api/sequences/create — create a sequence for a requirement with selected candidates
   - POST /api/sequences/{id}/generate — generate personalised emails for all candidates
   - GET /api/sequences — list all sequences with stats
   - GET /api/sequences/{id} — detail view with all emails
   - POST /api/sequences/{id}/approve — mark all draft emails as ready to send

5. No auto-send. Everything stays as drafts until explicitly approved. This is a hard rule.
```

**After Day 5:** Select candidates from search results → generate personalised outreach emails → review → approve → schedule. The full sourcing-to-outreach pipeline works.

### Day 6: Client Submissions + TL Approval (Shoham)

**The problem (from the competitive analysis):** This is where we beat everyone. Juicebox stops at outreach. Recruiterflow stops at outreach. Nobody builds client submission formatting with TL approval workflow.

**Prompt for Conductor:**
```
Build the client submission workflow:

1. ai-agents/formatter.py:
   - Function: format_submission(candidates: list[dict], requirement: dict, client: dict) -> dict
   - Takes shortlisted candidates and generates a formatted submission document:
     - Client name + requirement title at top
     - For each candidate: name, current role, experience summary, skills match (from scorer), AI recommendation, recruiter notes
     - Summary stats: total shortlisted, average match score, location breakdown
   - Return as HTML string (for email) and as structured JSON (for the dashboard)

2. Supabase: submissions table should have:
   - requirement_id, candidate_ids (array), formatted_html, status (draft/pending_tl/approved/sent_to_client/client_responded)
   - tl_notes, approved_by, approved_at, sent_at
   - client_feedback (JSON — per-candidate thumbs up/down + comments)

3. FastAPI endpoints:
   - POST /api/submissions/create — create submission from shortlisted candidates
   - POST /api/submissions/{id}/format — generate the formatted document
   - POST /api/submissions/{id}/approve — TL approves, status -> approved
   - POST /api/submissions/{id}/reject — TL rejects with notes
   - GET /api/submissions — list with status filters
   - GET /api/submissions/{id} — detail with formatted content

4. The TL Approvals page in the dashboard should show real submissions with Approve/Reject buttons that call these endpoints.

5. When TL approves, the submission is marked ready but NOT auto-sent. The recruiter manually sends it.
```

**After Day 6:** Full pipeline: Search → Score → Shortlist → Generate Submission → TL Approval → Send to Client. This is the agency workflow that Juicebox, Recruiterflow, and GoPerfect don't have.
