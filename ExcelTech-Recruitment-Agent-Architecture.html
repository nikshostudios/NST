SYSTEM DESIGN v2.1

The Recruitment
Agent Layer

A system design for automating the entire recruitment pipeline at ExcelTech — from requirement intake to client submission — using an army of AI agents. The single goal: double or triple recruiter throughput (from ~6 submissions/day to 12-20) by eliminating grunt work and multiplying sourcing reach. This document covers the enhanced multi-channel sourcing architecture, new agents, upgraded orchestration, and a modular BD add-on for agencies that want proactive business development.

Author: Shoham + Claude Date: 2026-04-13 Status: DESIGN Builds on: ExcelTech v1 (live in production)

00Where We Are Today

ExcelTech's v1 automation is live in production with 5 agents (Screener, Outreach, Follow-up, Formatter, Sourcing) and 5 skills. Recruiters went from 6-8 hours of mechanical work to ~1.5-2 hours. But sourcing is the weakest link:

BLOCKED
Foundit Scraping

Firecrawl hits Akamai bot detection. Cookie-based sessions expire every 24-48h. CAPTCHA blocks automated login refresh. The SPA API endpoint reverse-engineering approach is the right next step, but it's fragile.

WRONG DATA
MyCareersFuture

Returns job postings, not candidate profiles. Can't be upserted to candidates table. By design — MCF is a job board for job seekers, not a candidate sourcing database for recruiters.

WORKING
Internal DB

2,300+ candidates from prior placements. Zero cost. Sourcing agent checks this first. But it's a fixed pool — no new candidates enter unless sourced externally.

MANUAL ONLY
LinkedIn

Hard rule: no automation, no scraping. Recruiters manually search, receive CVs via DM. LinkedIn posts generate inbound. Non-negotiable for Phase 2 credibility.

THE CORE PROBLEM

ExcelTech has one unreliable external sourcing channel (Foundit via Firecrawl) and one working but fixed internal DB. If Foundit stays blocked, the automation layer can screen, outreach, follow-up, and format beautifully — but it has almost nothing to feed into the pipeline. Sourcing is the bottleneck, and sourcing is the biggest opportunity.

01The Enhanced Sourcing Architecture

The v2 design introduces a multi-channel sourcing mesh that eliminates single-channel dependency. Instead of relying on one fragile Foundit connection, we build a layered system with 8 sourcing channels — every one of them focused on finding candidate profiles for requirements that clients give us. The mission is simple: when a JD arrives, the system searches everywhere, enriches every result with direct contact info, and hands the recruiter a ready-to-call shortlist.

THE THROUGHPUT MULTIPLIER

v1: Client gives JD → Recruiter searches Foundit manually → Finds 5-8 candidates/day → Screens one by one → Submits ~6 profiles

v2: Client gives JD → 8 channels searched in parallel → AI screens + ranks 100+ candidates in minutes → Enriches with phone/email → Recruiter reviews shortlist → Submits 12-20 profiles/day

Channel 3: Apollo.io LIVE — UPGRADE
Global • Candidate Profiles + Contact Data • Already integrated in sourcing.py
What
275M+ verified contacts with direct emails, phone numbers, job titles, companies, tech stacks, and career history. Apollo is both a discovery engine (search for "ServiceNow Developer, 8+ years, Bangalore") and an enrichment tool (paste a LinkedIn URL, get the full contact card).
Why It Matters
Already proven in production. Nikhil used Apollo to find 30 fresh candidates for a senior sourcing manager role in Singapore after Foundit and MCF were exhausted. Unlike job boards where you're limited to active job seekers, Apollo indexes professionals who have never touched a job board — passive candidates who'd move for the right offer. Apollo finds them and gives you their direct phone number.

1. Direct candidate search: "ServiceNow ITSM, 8+ years, Bangalore, currently at Infosys/Wipro/TCS" → returns candidates with verified emails + phones
2. Enrichment: Candidate found on Foundit has no phone → Apollo lookup fills in direct mobile + personal email
3. Company-based sourcing: "Show me all DevOps engineers at HCL in Bangalore" → target specific talent pools
4. Lookalike search: "Find 20 people similar to this placed candidate" → replicate successful placements
Pricing
Free tier: 10,000 records/month (generous for start). Paid: $49/mo (Basic), $99/mo (Professional). API access on paid tiers.

GH
Channel 4: GitHub Talent Sourcing NEW
Global • Tech Candidates • Passive Talent
What
GitHub has 100M+ developers with public profiles, contribution histories, and tech stack fingerprints. By searching repos, commit activity, and profile bios, we can find developers by what they actually build, not just what they claim on a resume.
Why
For tech roles (DevOps, Data Engineering, Full Stack, Cloud), GitHub is the highest-signal channel. A developer with 500+ commits to ServiceNow-related repos in the last year is more qualified than someone who listed "ServiceNow" on their Naukri profile 3 years ago. This is proof of work, not self-reported skills.
Approach
GitHub API (free): Search users by location + language + bio keywords. Check contribution frequency, repo topics, and starred technologies.
Enrichment chain: GitHub profile → extract email from public commits (git log) → if no email, pass to Hunter.io/Apollo for enrichment → create candidate record.
Apify GitHub Scrapers: Pre-built actors for structured profile extraction if API rate limits are hit.
Cost
GitHub API: Free (5,000 requests/hour with auth token). Apify actors: ~$5-10/month if needed.
Best For
DevOps, Data Engineering, Full Stack, Cloud, Security roles. Not useful for non-tech roles (ServiceNow functional consultants, BAs).
Lu
Channel 5: Lusha (Primary) + Hunter.io (Secondary) NEW — TO BUILD
Global • Phone Numbers + Email Verification
Priority
Phone numbers are what recruiters actually need. Email is secondary — if a recruiter has a phone number, they can WhatsApp or call directly. That's how Indian and Singapore recruitment actually works. The enrichment priority is: phone first (Lusha → Apollo fallback), then email verification (Hunter.io).
Lusha (Phone)
Direct phone number finder. 100M+ business profiles. Given name + company (or LinkedIn URL), returns direct mobile number + personal email. This is the #1 enrichment tool for recruiters globally. Not yet implemented — needs to be built into sourcing.py.
Hunter.io (Email)
Email finder + verifier. 107M+ indexed addresses. Primary use: verify that emails from Foundit/Naukri/Apollo are still deliverable before Outreach Agent drafts anything. Secondary: find professional email when only name + company is known. Not yet implemented — needs to be built into sourcing.py.
Enrichment Pipeline
Built into the Sourcing Agent as a post-processing step — not a separate agent:
1. Candidate sourced from any channel (Foundit/Naukri/Apollo/GitHub/Inbound)
2. Missing phone? → Lusha lookup (name + company) → if Lusha fails → Apollo enrichment endpoint
3. Missing/unverified email? → Hunter.io verify → if no email → Hunter.io finder (name + company domain)
4. Cache all lookups in enrichment_cache table (avoid duplicate API calls, respect rate limits)
5. Candidate presented to recruiter with verified phone + email
Cost
Lusha: Free (50 credits/mo), Pro $49/mo (160 credits). Hunter.io: Free (25 searches/mo), Starter $49/mo (500 searches). Smart caching reduces lookups by ~40% (same candidate found across multiple channels only enriched once).
Implementation
Add enrich_candidate_contact() function to sourcing.py. Call Lusha API (api.lusha.com/person) and Hunter.io API (api.hunter.io/v2/email-finder). Both are simple REST APIs with API key auth. ~50 lines of code each.

Sc
Screener Agent ENHANCED
Sonnet 4
v1
Match score 0-100, risk flags, reasoning
v2: Salary Scoring (Fixed Logic)
Client budget is the primary reference. Market data is secondary.

Rule 1 — Client gave a budget (most agency work): If the candidate's expected CTC is within the client's stated budget, do NOT penalize the score — even if the candidate is asking above market rate. If the client says "budget is 25 LPA" and candidate asks 22 LPA, that's a perfect fit. Only penalize when candidate's ask exceeds the client's budget. Penalty scales: 5-10% over budget = minor flag, 10-20% = moderate, 20%+ = major red flag.

Rule 2 — No client budget (enterprise/internal TA): Fall back to Adzuna market benchmarks. Show "Above market by X%" as an informational flag, not a score penalty. These clients are usually flexible — the flag helps them negotiate, not reject.

Implementation: Update screener.md prompt to check requirements.salary_budget first. If present, use it as the ceiling. If null/empty, query market_salary_benchmarks for the role + location. Current screener uses a simple yes/no/unknown for salary_fit — upgrade to a numeric scale with reasoning.
v2: Multi-Req Matching
Multi-requirement matching. When a new candidate enters the system (via any channel), screen against ALL open requirements, not just the one being sourced. "This candidate was sourced for Req #45 but also matches Req #62 at 87%." Surfaces cross-selling opportunities — the single fastest way to increase submissions per day.

New Agents

JD
JD Parser Agent NEW
Sonnet 4
Problem
Skills not being extracted from JD text on requirement creation (current bug — possibly silent JSON parse failure). Recruiters paste JDs in wildly different formats. Skill normalization gap: "ServiceNow JavaScript ITSM" should become three separate skills.
Solution
Dedicated agent that runs on every new requirement. Extracts: required skills (normalized to canonical list), experience range, salary range, location, notice period tolerance, must-have vs nice-to-have, red flags (unrealistic combos like "10 years of ServiceNow with 3 LPA"). Outputs structured JSON that seeds the requirement record properly.
Trigger
Auto on requirement creation. Also available for re-parse when JD is updated.
Why Separate
JD parsing is a different skill from candidate screening. The Screener needs a well-parsed JD as input. If the JD is badly parsed, every downstream agent suffers. This agent is the quality gate at the top of the funnel.
Cost
~$0.01-0.03 per JD (Sonnet 4, ~1-2K tokens). 50 new reqs/month = ~$1.50/month.
Re
Reactivation Agent NEW
Haiku 4.5
Problem
ExcelTech has 2,300+ candidates in the DB but most are "cold" — sourced months ago, not contacted recently. Meanwhile, new requirements come in daily that might match these dormant candidates. Currently, no one tracks why a candidate was rejected — so when a new requirement comes in that would actually fit, nobody knows to look back.
Solution
Context-aware reactivation with rejection memory. Two parts:

Part 1 — Rejection Tracking (schema change): Add rejection_reason column to screenings table. When a candidate is rejected or deprioritized, record why: "salary too high (asked 25 LPA, budget was 18 LPA)", "notice period too long (90 days, client needed 30)", "location mismatch (Bangalore, role was Chennai)", "overqualified", "skills gap in X". This data becomes the Reactivation Agent's intelligence.

Part 2 — Smart Weekly Scan: For each open requirement, scans the full internal DB (vector search) for candidates who:
1. Were sourced 30+ days ago
2. Were rejected for a different role — and the rejection reason no longer applies
3. Match the current requirement at 70%+ confidence

Context-aware examples:
• "Rejected for Req #34 because salary too high at 25 LPA — but Req #67 has 28 LPA budget" → RESURFACE
• "Rejected for Req #40 because notice period was 90 days — Req #67 has flexible start date" → RESURFACE
• "Rejected for Req #34 because skills gap in Kubernetes — Req #67 also needs Kubernetes" → SKIP

Generates a "Reactivation Queue" with context: "These 8 candidates from your old pipeline might fit Req #67. Here's why they were rejected before and why it doesn't apply now."
Why
Reactivation is a $0 sourcing cost channel. These candidates already have CVs parsed, contact info confirmed, sometimes even field data from prior outreach. The fastest path to a placement is often someone already in the system. Context-aware resurfacing means recruiters don't waste time re-reviewing candidates who'd be rejected for the same reason.
Trigger
APScheduler: weekly Sunday night. Also on-demand when any new requirement is created — the first thing the system does before external sourcing is check if there's a match sitting dormant in the DB.
Implementation
Schema: Add rejection_reason TEXT and rejection_category TEXT (salary/notice/location/skills/overqualified/other) to screenings table. Agent: Prompt Haiku to compare rejection_reason against new requirement's budget, timeline, and location to determine if the blocker still applies. Route: Existing /candidates/reactivation-scan endpoint needs upgrade. Effort: ~2-3 hours of work.

08Trade-offs & Risks

RISK
Foundit/Naukri API Fragility

Reverse-engineered APIs can break when the platform updates their SPA. Mitigation: Monitor response shapes with schema validation. Alert on breakage within 15 min. Bright Data as fallback. Never depend on a single channel.

RISK
Cookie/Session Expiry

Automated login via Playwright can be detected. Mitigation: Run from residential IP (Bright Data proxies if Railway IP gets blocked). Randomize login timing. Only 3-4 logins/day total.

TRADE-OFF
Complexity vs Reliability

8 channels = 8 things that can break. Decision: Each channel has an independent health check and can be disabled without affecting others. The system degrades gracefully — if 3 channels are down, the other 5 still work.

TRADE-OFF
Enrichment API Costs Scale With Volume

Apollo + Hunter + Lusha credits are consumed per lookup. At high volume (100+ candidates/day), costs scale. Mitigation: Enrichment cache avoids duplicate lookups. Only enrich candidates who pass initial Screener threshold (score 60+). Apollo free tier covers 10K records/month — substantial for early growth.

OPEN QUESTION
Naukri Enterprise API Approval

Naukri may take weeks to approve enterprise API access. Contingency: Start with Apify Naukri actor immediately. Switch to enterprise API when approved. Same output schema either way.

OPEN QUESTION
Embedding Model Choice

Claude doesn't have a dedicated embeddings endpoint. Options: Use Voyage AI (Anthropic-recommended), OpenAI text-embedding-3-small ($0.00002/1K tokens), or Cohere embed-v3. All produce 1536-dim vectors. Voyage likely best for Claude-ecosystem coherence.

08bAdditional Feature Recommendations

Based on the codebase audit and recruitment domain knowledge, these features would materially increase the product's value. Ordered by impact × implementation effort.

HIGH IMPACT / LOW EFFORT
WhatsApp Integration via Twilio/WhatsApp Business API

You said it yourself — recruiters WhatsApp candidates first before emailing. Right now that's fully manual. Add WhatsApp template messaging: after a candidate is shortlisted, auto-draft a WhatsApp message (interest check) that the recruiter sends with one click. Same pattern as the Outreach Agent (draft, review, send). Uses WhatsApp Business API via Twilio (~$0.05/message). This alone could save 30+ minutes/day per recruiter.

Feasibility: Twilio WhatsApp API is well-documented, REST-based, ~100 lines of Python. Needs WhatsApp Business account verification (1-2 weeks). Message templates need Meta approval. Effort: 1 week.

HIGH IMPACT / LOW EFFORT
CV Auto-Parser (Resume → Structured Data)

When a candidate sends their CV (via email, WhatsApp, or inbound portal), auto-extract all fields using Claude: name, skills, experience, education, current CTC, notice period, certifications. Pre-fill the candidate_details record. Recruiter only needs to verify, not type. Saves the entire "candidate fills in the table" step for candidates who already have a detailed CV.

Feasibility: Claude can parse PDFs and DOCXs natively. The Formatter Agent already has the output schema. Reverse it: CV → structured JSON → upsert to candidate_details. Effort: 3-4 hours.

HIGH IMPACT / MEDIUM EFFORT
Duplicate Candidate Detection Across Requirements

When the same candidate appears from multiple channels or across multiple requirements, merge intelligently instead of creating duplicates. Current dedup is email-first, name+employer fallback. Upgrade to: phone match, LinkedIn URL match, and fuzzy name matching (Jaro-Winkler similarity). Surface to recruiter: "This candidate was already contacted for Req #34 — here's their status."

Feasibility: Most dedup logic exists in sourcing.py. Adding phone + LinkedIn + fuzzy match is ~50 lines. Schema already has linkedin_url column. Effort: 1-2 days.

MEDIUM IMPACT / MEDIUM EFFORT
Candidate Availability Tracker

Many candidates say "I'm interested but my notice period ends in 60 days." The system should auto-schedule a follow-up for Day 45, auto-draft a "checking in" message, and auto-match them against any requirements open at that future date. Turns "not now" into "scheduled placement."

Feasibility: candidate_details.availability_date already exists. APScheduler already runs. Add a daily scan: candidates whose availability_date is within 15 days → auto-match against open reqs → notify recruiter. Effort: 1 day.

MEDIUM IMPACT / LOW EFFORT
Recruiter Performance Dashboard

Track per-recruiter metrics: candidates sourced, emails sent, replies received, submissions made, placements closed. Data already exists in the DB — it just needs a dashboard view. Shows management which recruiters are productive and where the pipeline is bottlenecked.

Feasibility: All data lives in outreach_log, submissions, screenings. Dashboard component in React. SQL aggregation queries. Effort: 2-3 days.

FUTURE / HIGH EFFORT
Voice Agent for Initial Candidate Screening Calls

An AI voice agent that makes initial interest-check calls to shortlisted candidates: "Hi, I'm calling from ExcelTech about a ServiceNow opportunity in Bangalore. Are you open to hearing more?" Captures yes/no/callback time. Replaces the most time-consuming recruiter task — the initial phone screen. Uses Twilio + Vapi/Retell AI. This is Phase 3 territory but could be a massive differentiator.

Feasibility: Vapi.ai and Retell.ai both offer production-grade voice agents with API. Cost: ~$0.10-0.20/min. Needs careful compliance review for automated calling laws in India/SG. Effort: 2-3 weeks. Phase 3.
