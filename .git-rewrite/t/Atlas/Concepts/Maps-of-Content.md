# Maps of Content (MOCs)

> "A MOC isn't a table of contents. It's an invitation."
> — Nick Milo

## The idea in plain English

A Map of Content (MOC) is a note whose job is to *point to other notes and make them useful together*. It's the Obsidian/PKM answer to folders. Instead of organizing by filesystem hierarchy, you organize by meaning — one note that curates a topic and links outward in whatever shape makes sense.

The critical distinction is **active vs. passive**:

- A **passive** MOC is a list of links. You read it once, click one, and never come back.
- An **active** MOC curates. It groups related notes, surfaces the most important ones first, calls out open threads, and changes over time. You *return* to it because it does work for you.

Every good MOC answers a question. If someone opened this MOC cold, what would they be looking for? What would they leave with?

## Why it matters for Niksho

- **Our three headspace MOCs — [[Atlas/Atlas]], [[Calendar/Calendar]], [[Efforts/Efforts]] — are the main navigation layer of the vault.** If these are passive, the vault is a warehouse. If they're active, the vault is a workshop.
- **AI sessions benefit enormously from good MOCs.** When Claude opens [[Home]] and drills into [[Atlas/Atlas]], an active MOC that says "start with these three notes" saves 10 tool calls worth of searching.
- **Onboarding (human or AI) collapses to MOC-reading.** A newcomer should be able to get 80% of our context by reading 5-10 notes — the MOCs + the top entries they point to.

## Key moves

1. **Start with the question.** Before writing a MOC, write down: "if I opened this cold, what am I trying to find?" The answer shapes everything.
2. **"Start here" first.** Every MOC should have a "🌟 Start here" section with 3-5 notes. This is the unofficial contract: *if you read only these, you'll understand the area*.
3. **Group by meaning, not alphabet.** The sections of a MOC should tell a story. Alphabetical order is what you use when you've given up on curation.
4. **Include open threads.** What's unresolved in this area? What would you chase with an afternoon? This turns the MOC into a TODO for your future self.
5. **Revisit on schedule.** A MOC that was written once and never updated is a tombstone. Weekly reviews should touch at least one MOC.

## The three kinds of MOCs we use

| Kind | Example | Purpose |
|------|---------|---------|
| **Headspace MOC** | [[Atlas/Atlas]], [[Calendar/Calendar]], [[Efforts/Efforts]] | Top-level navigation for a headspace |
| **Topic MOC** | Could be `Atlas/Product/Product-MOC.md` or a curated view of ExcelTech | A lens into one domain |
| **Question MOC** | "What do we actually know about the SG market?" | Answers a specific recurring question |

## Common failure modes

- **The index trap.** A MOC that's just `- Note A  - Note B  - Note C` is an index, not a MOC. Add structure, add a "start here", add context sentences.
- **Set-and-forget.** MOCs rot faster than atomic notes because they're inherently about the current shape of your thinking. If you don't touch them, they lie.
- **MOC sprawl.** Too many MOCs and they start competing with each other. Start with the three headspace MOCs and add new ones only when a real recurring question demands one.
- **Confusing MOC with dashboard.** [[Home]] is a dashboard. It gives you the current state of the business. A MOC is about a topic area. Don't merge them.
