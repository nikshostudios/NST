## The original Karpathy tweet (verbatim)

> **LLM Knowledge Bases**
>
> Something I'm finding very useful recently: using LLMs to build personal knowledge bases for various topics of research interest. In this way, a large fraction of my recent token throughput is going less into manipulating code, and more into manipulating knowledge (stored as markdown and images). The latest LLMs are quite good at it. So:
>
> **Data ingest:** I index source documents (articles, papers, repos, datasets, images, etc.) into a `raw/` directory, then I use an LLM to incrementally "compile" a wiki, which is just a collection of `.md` files in a directory structure. The wiki includes summaries of all the data in `raw/`, backlinks, and then it categorizes data into concepts, writes articles for them, and links them all. To convert web articles into `.md` files I like to use the Obsidian Web Clipper extension, and then I also use a hotkey to download all the related images to local so that my LLM can easily reference them.
>
> **IDE:** I use Obsidian as the IDE "frontend" where I can view the raw data, the compiled wiki, and the derived visualizations. Important to note that the LLM writes and maintains all of the data of the wiki, I rarely touch it directly. I've played with a few Obsidian plugins to render and view data in other ways (e.g. Marp for slides).
>
> **Q&A:** Where things get interesting is that once your wiki is big enough (e.g. mine on some recent research is ~100 articles and ~400K words), you can ask your LLM agent all kinds of complex questions against the wiki, and it will go off, research the answers, etc. I thought I had to reach for fancy RAG, but the LLM has been pretty good about auto-maintaining index files and brief summaries of all the documents and it reads all the important related data fairly easily at this ~small scale.
>
> **Output:** Instead of getting answers in text/terminal, I like to have it render markdown files for me, or slide shows (Marp format), or matplotlib images, all of which I then view again in Obsidian. You can imagine many other visual output formats depending on the query. Often, I end up "filing" the outputs back into the wiki to enhance it for further queries. So my own explorations and queries always "add up" in the knowledge base.
>
> **Linting:** I've run some LLM "health checks" over the wiki to e.g. find inconsistent data, impute missing data (with web searches), find interesting connections for new article candidates, etc., to incrementally clean up the wiki and enhance its overall data integrity. The LLMs are quite good at suggesting further questions to ask and look into.
>
> **Extra tools:** I find myself developing additional tools to process the data, e.g. I vibe coded a small and naive search engine over the wiki, which I both use directly (in a web UI), but more often I want to hand it off to an LLM via CLI as a tool for larger queries.
>
> **Further explorations:** As the repo grows, the natural desire is to also think about synthetic data generation + finetuning to have your LLM "know" the data in its weights instead of just context windows.
>
> **TLDR:** raw data from a given number of sources is collected, then compiled by an LLM into a `.md` wiki, then operated on by various CLIs by the LLM to do Q&A and to incrementally enhance the wiki, and all of it viewable in Obsidian. You rarely ever write or edit the wiki manually, it's the domain of the LLM. I think there is room here for an incredible new product instead of a hacky collection of scripts.

## Related community implementation — `claude-memory-compiler`
**Repo:** https://github.com/coleam00/claude-memory-compiler
**Author:** Cole Medin (coleam00)

This is the self-evolving Claude Code memory system built on top of Karpathy's architecture, but applied to **internal** data (session logs from Claude Code conversations) rather than external data (articles, papers).

Key ideas from the repo and its companion video (see [[Raw/transcripts/YT-Transcripts-2026-04-10]] Tab 4):

- **Compiler analogy.** raw → compile → wiki, mirroring source code → compile → executable.
- **Session logs = raw folder.** Claude Code hooks capture conversation summaries automatically.
- **Wiki = concepts + connections + index + agents.md** — one `agents.md` that describes the whole system to any coding agent dropped into the project.
- **Three hooks:**
  - `session-start` — loads `agents.md` and `wiki/index.md` into the fresh session
  - `pre-compact` — summarises recent messages before context is about to be lost
  - `session-end` — writes the summary to the daily log file
- **Daily flush** — once a day, extract concepts & connections from the logs and promote to the wiki.
- **Compounding loop.** Every answer the agent gives creates new links which become new queryable knowledge. The wiki gets smarter over time without manual maintenance.

This repo is the closest reference implementation to what we want for **Niksho's second brain**, with one adjustment: we blend Karpathy's raw→wiki external-research pattern with Nick Milo's ACE authored-knowledge pattern, rather than picking one. See [[Atlas/Concepts/AI-OS]] for our hybrid reasoning.

## Why this matters for Niksho

1. **Token economics.** One cited result: 383 scattered files + 100 meeting transcripts → compacted wiki → **95% token reduction** when querying with Claude. This is the difference between a vault we can afford to query daily and one we can't.
2. **Portability.** It's just markdown files. No vector DB, no embedding model, no reindexing pipeline. Matches our [[Atlas/Concepts/File-over-AI]] principle.
3. **Works at our scale.** Karpathy says ~100 articles / ~400k words is fine with index + follow-links navigation. We are nowhere near that ceiling and we can scale the Niksho brain for years before we need anything fancier.
4. **Fits our hybrid.** The raw/wiki compiler pattern is orthogonal to the authored Atlas/Calendar/Efforts — they coexist. See [[AIOS/vault-map]] for the full layout.
5. **The same pattern scales to the product.** When we productise for other agencies (see [[Atlas/Business-Model/Phase-2-GTM]]), each agency gets its own vault with its own raw/ and wiki/. The operating system is transplantable.
